# 단골만들기 1차 MVP 구현 지시서

## 0. 문서 목적

이 문서는 `단골만들기` 1차 MVP를 클로드(Claude)에게 구현 요청하기 위한 최종 스펙 문서이다.

이번 MVP는 완성형 CRM이 아니라, 아래 핵심 자동화 루프가 실제로 작동하는지 검증하는 제품이다.

```text
QR로 고객을 모은다
→ 방문 데이터를 쌓는다
→ 고객 상태에 맞는 메시지를 만든다
→ 카카오톡으로 자동 발송한다
→ 재방문을 유도한다
```

---

## 1. 서비스 한 줄 정의

**단골만들기는 점주가 QR로 고객 정보를 모으면, 고객 방문 기록을 기반으로 AI/템플릿 메시지를 생성하고 카카오톡으로 자동 발송해 재방문을 유도하는 소상공인용 리텐션 SaaS이다.**

---

## 2. 핵심 용어 정의

| 용어 | 의미 |
|---|---|
| 점주 | 단골만들기를 사용하는 소상공인 / 매장 운영자 / 결제 주체 |
| 고객 | 점주의 손님 / QR로 등록하는 사람 / 단골 후보 |
| 관리자 | 단골만들기 서비스 운영자 |

코드 기준 추천 용어:

| 개념 | 추천 테이블/모델명 |
|---|---|
| 점주/매장 | stores 또는 merchants |
| 고객 | customers |
| 방문 기록 | visit_logs |
| 메시지 기록 | message_logs |
| 관리자 | admins |

---

## 3. MVP 핵심 원칙

### 3-1. 점주의 역할은 최소화한다

점주의 기본 역할은 아래 2가지다.

```text
매장에 QR을 비치한다.
고객에게 QR 등록을 유도한다.
```

점주가 매번 메시지를 작성하거나, AI가 만든 문구를 복사하거나, 카카오톡으로 직접 발송하는 구조가 아니다.

---

### 3-2. 카카오톡 자동발송은 1차 MVP 핵심 기능이다

단골만들기의 핵심 가치는 자동화다.

따라서 1차 MVP에는 카카오톡 자동발송이 반드시 포함되어야 한다.

다만 1차에서는 고급 발송 운영 기능까지 만들 필요는 없다.

| 포함 | 제외 |
|---|---|
| 최소 카카오 발송 API 연동 | 고급 발송 리포트 |
| 신규 등록 감사 메시지 자동발송 | 복잡한 캠페인 설정 |
| 재방문 유도 메시지 자동발송 | 링크 클릭 추적 |
| 휴면 고객 리마인드 메시지 자동발송 | 쿠폰 시스템 |
| 발송 성공/실패 로그 저장 | 발송 비용 상세 분석 |
| 단순 재시도 또는 실패 기록 | 정교한 재시도 큐 |

---

### 3-3. 고객 식별은 휴대폰 번호 기반으로 한다

1차 MVP에서는 카카오 인증 기반 식별을 사용하지 않는다.

고객 식별 기준:

```text
store_id + phone
```

같은 휴대폰 번호라도 매장이 다르면 다른 고객으로 본다.

---

## 4. 1차 MVP 필수 기능

1차 MVP의 필수 기능은 아래 8개다.

| 순서 | 기능 | 목적 |
|---:|---|---|
| 1 | 점주/매장 생성 | 서비스 사용 주체 생성 |
| 2 | 매장별 QR URL 생성 | 고객 등록入口 확보 |
| 3 | 고객 휴대폰 번호 기반 등록 | 고객 데이터 확보 |
| 4 | 신규/기존 고객 자동 분기 | 등록과 재방문을 하나의 QR로 처리 |
| 5 | 기존 고객 방문 체크인 | 재방문 기록 확보 |
| 6 | 방문주기 단순 계산 | 다음 발송 타이밍 계산 |
| 7 | 카카오톡 자동발송 | 핵심 자동화 가치 검증 |
| 8 | 최소 관리자 페이지 | 전체 운영 상태 확인 |

---

## 5. QR 코드 구조

매장마다 하나의 QR을 발급한다.

예시 URL:

```text
https://dangol.app/s/{store_id}
```

QR의 역할은 고객 개인 식별이 아니라 `매장 식별`이다.

고객 개인 식별은 QR 접속 후 입력하는 휴대폰 번호로 처리한다.

---

## 6. 신규 고객 / 기존 고객 자동 분기

고객이 QR을 스캔하면 아래 흐름으로 처리한다.

```text
QR 스캔
→ store_id 확인
→ 휴대폰 번호 입력
→ DB에서 store_id + phone 조회
→ 등록 이력 없음: 신규 고객 등록
→ 등록 이력 있음: 기존 고객 방문 체크인
```

---

## 7. 신규 고객 등록 흐름

### 7-1. 고객 화면 흐름

```text
QR 스캔
→ 휴대폰 번호 입력
→ 등록 이력 없음
→ 이름 입력
→ 개인정보 수집 동의
→ 마케팅 수신 동의
→ 등록 완료
→ 첫 방문 기록 생성
→ 신규 등록 감사 카카오톡 자동발송
```

### 7-2. 신규 고객 등록 시 저장할 정보

| 필드 | 설명 |
|---|---|
| store_id | 어느 매장의 고객인지 |
| name | 고객 이름 |
| phone | 고객 휴대폰 번호 |
| privacy_agreed | 개인정보 동의 여부 |
| marketing_agreed | 마케팅 수신 동의 여부 |
| created_at | 등록일 |
| last_visit_at | 마지막 방문일 |
| visit_count | 방문 횟수 |

1차 MVP에서는 생일, 성별, 관심사 등 추가 정보는 받지 않는다.

---

## 8. 기존 고객 방문 체크인 흐름

### 8-1. 기존 고객 화면 흐름

```text
QR 스캔
→ 휴대폰 번호 입력
→ 등록 이력 있음
→ “오늘 방문 확인” 화면
→ 방문 체크인 완료
→ 방문 기록 추가
→ 고객 방문 정보 갱신
```

### 8-2. 방문 체크인 시 갱신할 정보

| 필드 | 처리 |
|---|---|
| visit_logs | 방문 기록 추가 |
| customers.last_visit_at | 오늘 날짜로 갱신 |
| customers.visit_count | +1 |
| customers.average_visit_cycle_days | 단순 평균으로 재계산 |
| customers.next_expected_visit_at | 마지막 방문일 + 평균 방문주기 |

---

## 9. 방문주기 계산

1차 MVP에서는 복잡한 예측 모델을 쓰지 않는다.

단순 평균 방식으로 계산한다.

```text
방문주기 = 최근 방문 간격들의 평균
다음 예상 방문일 = 마지막 방문일 + 평균 방문주기
```

방문 횟수별 처리:

| 방문 횟수 | 처리 |
|---:|---|
| 1회 | 신규 고객 |
| 2회 | 첫 재방문 고객, 첫 방문 간격 계산 |
| 3회 이상 | 평균 방문주기 계산 |
| 장기간 미방문 | 휴면 후보 |

1차 MVP의 고객 상태값은 아래 3개로 충분하다.

```text
new
active
dormant
```

## 10. 점주 수동 방문 보정

### 10-1. 포함 여부

점주 수동 방문 보정은 1차 MVP에 포함한다.

단, 기본 기능이 아니라 `누락 보정용 보조 기능`으로만 둔다.

기본 방문 갱신은 고객 QR 체크인이다.

---

### 10-2. 필요한 이유

현장에서는 아래 상황이 발생할 수 있다.

| 케이스 | 설명 |
|---|---|
| 고객이 QR 체크인을 하지 않았지만 실제 방문 | 가장 흔한 누락 케이스 |
| 고객이 휴대폰 번호를 잘못 입력 | 중복 고객 생성 가능 |
| 고객이 휴대폰 번호를 변경 | 기존 이력이 끊길 수 있음 |
| 가족/대리 방문 | 등록자와 실제 방문자가 다를 수 있음 |

---

### 10-3. 1차 MVP에 넣을 최소 기능

점주 화면에는 아래 기능만 넣는다.

| 기능 | 설명 |
|---|---|
| 고객 검색 | 이름 또는 휴대폰 번호로 검색 |
| 오늘 방문 처리 | 오늘 날짜 기준으로 방문 기록 추가 |
| 고객 정보 수정 | 이름, 휴대폰 번호 수정 |

아래 기능은 1차 MVP에서 제외한다.

| 제외 기능 | 처리 |
|---|---|
| 과거 날짜 방문 추가 | 2차 |
| 고객 메모 | 2차 |
| VIP 태그 | 2차 |
| 쿠폰 사용 처리 | 2차 |
| 중복 고객 병합 | 관리자 수동 대응 또는 2차 |

---

### 10-4. 수동 방문 처리 시 DB 처리

```text
점주 고객 검색
→ 고객 선택
→ 오늘 방문 처리
→ visit_logs 생성
→ source = manual
→ customers.last_visit_at 갱신
→ customers.visit_count 증가
→ average_visit_cycle_days 재계산
→ next_expected_visit_at 갱신
```

## 13. 카카오톡 자동발송 구조

### 13-1. 발송 대상

1차 MVP에서는 아래 3가지 메시지만 자동발송한다.

| 메시지 타입 | 발송 조건 | 목적 |
|---|---|---|
| welcome | 신규 고객 등록 직후 | 등록 감사 / 첫 혜택 안내 |
| revisit | 다음 예상 방문일 전후 | 재방문 유도 |
| dormant | 일정 기간 미방문 | 휴면 방지 |

---

### 13-2. 메시지 생성 방식

1차 MVP에서는 고급 AI 개인화보다 구조 검증이 우선이다.

따라서 아래 방식 중 하나로 구현한다.

```text
1순위: 템플릿 기반 메시지
2순위: 간단한 AI 생성 메시지
```

처음부터 복잡한 AI 프롬프트 관리 시스템은 만들지 않는다.

---

### 13-3. 발송 처리 흐름

```text
발송 대상 고객 조회
→ 메시지 타입 결정
→ 메시지 내용 생성
→ message_logs에 pending 저장
→ kakao_send_message() 호출
→ 성공 시 send_status = success
→ 실패 시 send_status = failed, error_message 저장
```

---

### 13-4. 카카오 발송 연동 기준

1차 MVP에는 최소 수준의 카카오톡 자동발송을 포함한다.

다만 아래 고급 기능은 제외한다.

| 포함 | 제외 |
|---|---|
| 카카오 발송 API 호출 | 복잡한 캠페인 빌더 |
| 발송 성공/실패 저장 | 고급 발송 통계 |
| 실패 사유 저장 | 비용 상세 리포트 |
| 기본 메시지 템플릿 | 쿠폰/링크 추적 |

카카오 발송 방식은 개발 환경에 맞춰 아래 중 하나를 선택한다.

```text
카카오 알림톡
카카오 친구톡
카카오 비즈메시지 대행사 API
```

## 15. 운영 통제용 최소 관리자 페이지

관리자 페이지는 포함한다.

다만 고급 분석용이 아니라, 전체 운영이 제대로 돌아가는지 확인하는 관제 화면이다.

### 15-1. 관리자 페이지 최소 메뉴

| 메뉴 | 기능 |
|---|---|
| 전체 현황 | 총 점주 수, 총 고객 수, 오늘 등록 수, 오늘 방문 수, 오늘 발송 수 |
| 점주 관리 | 점주/매장 목록, 구독 상태 확인 |
| 고객 현황 | 매장별 고객 수, 최근 등록 고객 확인 |
| 방문 로그 | QR 체크인/수동 방문 기록 확인 |
| 메시지 로그 | 생성 메시지, 카카오 발송 상태 확인 |

### 15-2. 관리자 페이지에서 제외할 것

| 제외 기능 | 처리 |
|---|---|
| 매출 리포트 | 2차 |

## 19. 클로드에게 전달할 핵심 지시문

아래 내용을 클로드에게 그대로 전달하면 된다.

```text
단골만들기 1차 MVP를 만들어줘.

이 MVP는 완성형 CRM이 아니라, 점주가 QR로 고객 정보를 모으고, 고객의 재방문을 기록하며, 방문주기를 기반으로 카카오톡 메시지를 자동발송하는 제품이야.

핵심 용어는 점주와 고객이야.
점주는 서비스를 사용하는 소상공인/매장 운영자이고,
고객은 점주의 매장에 방문해 QR로 등록하는 손님이야.

1차 고객 식별 방식은 휴대폰 번호 기반이야.
매장 QR은 store_id를 식별하고, 고객은 QR 접속 후 휴대폰 번호를 입력해.
DB에서 store_id + phone 조합으로 기존 고객 여부를 확인해야 해.

등록 이력이 없으면 신규 고객 등록 화면으로 보내고,
등록 이력이 있으면 기존 고객 방문 체크인으로 처리해.

기존 고객이 방문 체크인을 하면 visit_logs에 방문 기록을 추가하고,
customers의 last_visit_at, visit_count, average_visit_cycle_days, next_expected_visit_at을 갱신해야 해.

점주 수동 방문 보정 기능도 최소한으로 넣어줘.
단, 기본 기능이 아니라 고객이 QR 체크인을 하지 않았을 때 보정하는 용도야.
점주 화면에서는 고객 검색, 고객 상세, 오늘 방문 처리, 고객 정보 수정 정도만 있으면 돼.

카카오톡 자동발송은 1차 MVP 핵심 기능이야.
신규 등록 고객에게 welcome 메시지를 자동발송하고,
방문주기 기준으로 revisit 메시지,
장기 미방문 고객에게 dormant 메시지를 자동발송하는 구조를 만들어줘.

메시지는 처음에는 템플릿 기반 또는 간단한 AI 생성 방식으로 구현해도 돼.
중요한 것은 메시지 생성 → message_logs 저장 → 카카오 발송 API 호출 → 성공/실패 저장 흐름이 실제로 작동하는 거야.

점주 대시보드는 내 매장 QR, 고객 목록, 고객 상세, 오늘 방문 처리, 메시지 이력 정도로 단순하게 구성해줘.

관리자 페이지는 운영 통제용 최소 기능만 넣어줘.
전체 현황, 점주 관리, 고객 현황, 방문 로그, 메시지 로그만 있으면 돼.

핵심 과금모델은 월 34,800원이야.
48,600원은 점주 수 1,000개 이후 검토할 가격이므로 1차 MVP 기준은 34,800원으로 봐.
```

## 20. 최종 1차 MVP 요약

```text
점주가 매장을 만든다.
매장 QR이 생성된다.
고객이 QR을 찍고 휴대폰 번호로 등록한다.
기존 고객은 QR 재스캔 시 방문 체크인된다.
방문 기록이 쌓인다.
방문주기가 단순 계산된다.
고객 상태에 따라 메시지가 생성된다.
카카오톡으로 자동발송된다.
발송 성공/실패 로그가 저장된다.
점주는 누락 방문만 수동 보정한다.
관리자는 전체 운영 상태를 확인한다.
```

이 범위가 단골만들기 1차 MVP의 적정 범위이다.
